§1

What is turbopuffer?

turbopuffer is an S3-backed vector database built around SPFresh centroid-based ANN search. It stores the vector index on object storage and uses a local NVMe cache to accelerate repeated queries. It offers two deployment modes — serverless and pinned — and deliberately exposes no recall tuning knobs.

  • No recall tuning. Locked at 88.94% recall on filtered search and 98.51% on unfiltered. You cannot trade latency for recall.
  • Serverless mode. Shared compute, scales to zero between queries, inherent noisy-neighbor risk when multiple namespaces share a host.
  • Pinned mode. Dedicated NVMe, fixed replicas, full machine isolation. Eliminates noisy-neighbor contention but removes scale-to-zero savings.
  • Cold-start penalty. After any restart, the first queries must fetch index data from S3 before NVMe is warm. Penalty: 100–1500ms on first queries, converging to steady-state after ~30 requests.
  • SPFresh algorithm. Centroid-based ANN that post-filters for attribute predicates rather than integrating filters into graph traversal. Leads to measurable recall drops under selective filter conditions.

The multi-tenant dataset is 1M vectors total (100 tenants × 10K each) — 10× more data than DBpedia. Qdrant uploads all 1M sequentially in batches to one collection (payload_m=16, is_tenant=True), building per-tenant HNSW sub-graphs in the background. turbopuffer uploads 100 namespaces in parallel (asyncio.gather, semaphore=10 concurrent), so its 2.4 min wall-clock reflects parallel throughput across all tenants simultaneously. At comparable per-batch throughput (816 vs 6,847 wps), the parallel vs sequential methodology plus 10× more data explains the 8.5× wall-clock gap — not a Qdrant limitation per se.
Write-time vs query-time: turbopuffer stores 0.615 GB for DBpedia (≈ raw vector size: 100K × 1536-dim × 4B). Qdrant builds HNSW at write time — 195.8s for H&M, ~0s for DBpedia (concurrent), 5.1s for MT sub-graphs. This one-time write cost is the source of Qdrant’s 1.9ms query server latency: the index is in RAM, already built.

Why 100% recall? With ~10,000 vectors per tenant, both HNSW (Qdrant: m=0, payload_m=16 builds per-tenant sub-graphs) and SPFresh centroid structures (tpuf) have negligible approximation error — the per-tenant index is small enough for near-exhaustive traversal. p=32 autoscaling crossover: tpuf outperforms the single-node Qdrant at p=32 (16.8 vs 9.5 RPS) because Qdrant’s 2CPU node saturates (mean latency 5ms→105ms = 20×). tpuf autoscales horizontally. This is a node-sizing issue: a larger Qdrant instance restores the 5× advantage at all concurrency levels.
Billed GB per MT query: tpuf bills 0.256 GB per MT query vs ~0.031 GB raw vector data per namespace (10K × 768-dim × 4B) — ~8× overhead from centroid tree structure.

8.4 — Server-Side Latency Gap

Measuring server-reported latency (excluding network round-trip) reveals an architectural floor difference. At steady state on DBpedia (unfiltered), turbopuffer server latency is consistently ~10ms while Qdrant server latency is ~1.9ms.

Qdrant server latency is 1.9ms (HNSW index in RAM); tpuf server latency floor is 10ms (NVMe centroid traversal). This 5× gap is architectural and irreducible regardless of network. The gap reflects the fundamental storage medium: HNSW traversal on RAM vs. SPFresh centroid reads from NVMe. No configuration on either side can close it.

Pinned Mode: The Real Cost of Filtered Search

turbopuffer requires pinning for reliable filtered-search latency. The serverless model's scale-to-zero economics do not apply when the namespace must stay resident. Pinned billing introduces a 64 GB floor regardless of actual data size — with catastrophic consequences for small datasets.

9.1 — Billing Model

Pinned namespaces are billed at $0.01325/GB-hr (derived from turbopuffer's published example: $2,476 ÷ (256 GB × 730 hr/month)), applied to max(actual_gb, 64) × replicas × 730 hr/month. Storage is billed separately at $0.33/GB/month on actual data.

The 64 GB billing floor dominates everything below ~5 GB. For the H&M benchmark dataset (0.43 GB actual), pinned 4-replica costs $2,476/month — 95× more than Qdrant's $26.10 — while delivering worse throughput (6.0 RPS warm vs 159.4 RPS), worse p99 (538ms warm vs 10.1ms), and lower recall (88.94% vs 95.71%). For datasets under ~5 GB, pinning is strictly worse on every dimension: cost, throughput, and recall.

9.2 — Break-Even Analysis

For pinned 1-replica to match Qdrant's $26.10/month node cost, actual data must exceed 2.7 GB (so billing escapes the 64 GB floor and the compute cost drops below $26.10). That is approximately:

  • ~900K vectors at 1536-dim
  • ~1.8M vectors at 768-dim
  • ~350K vectors at 2048-dim (e.g. OpenAI text-embedding-3-large)

Below these sizes, Qdrant is cheaper even at zero QPS. At 4 replicas (required for reliable filtered search in production), there is no break-even at all — 4 replicas is the same data replicated, not 4× more data capacity. The minimum 4r cost ($2,476) always exceeds Qdrant's $26.10 regardless of dataset size.

The "~10 QPS cost crossover" applies only to serverless mode. When filtered search or isolation requirements force pinning, that crossover disappears entirely. For pinned 1r, break-even requires >2.7 GB actual data. For pinned 4r, there is no break-even — $2,476/month minimum vs $26.10.

Strategic Implications

These results point to specific architectural decisions rather than a simple winner. The right choice is a function of your QPS profile, filter selectivity, tenant concurrency pattern, dataset size, and tolerance for cold-start latency.

  1. Below ~10 QPS/namespace, serverless turbopuffer saves money. Above it, Qdrant is both cheaper and faster. The break-even is stable across dataset scales — plan around it. This only applies to serverless mode.
  2. Filtered search requires pinning, and pinning has a 64 GB billing floor. For any dataset under ~10 GB, pinned turbopuffer costs 10–95× more than Qdrant Cloud with no performance benefit. The serverless cost crossover does not apply to filtered search workloads.
  3. Serverless noisy-neighbor is a real operational risk. A single bursty tenant in serverless mode triples all other tenants' p50 latency. Pinning eliminates this but removes scale-to-zero. Design your tenancy model before choosing serverless.
  4. Filtered search recall is not tunable on turbopuffer. If your workload uses selective attribute filters and 88.94% recall is insufficient, Qdrant is the only option. This constraint is architectural, not a configuration gap.
  5. Multi-tenant idle SaaS is turbopuffer's strongest case. If most tenants are mostly idle, scale-to-zero is worth real money. At active-tenant load, Qdrant is 5.5× faster (184.6 vs 33.5 RPS) — the advantage flips completely when tenants are active.
  6. Cold-start latency must be part of your SLA calculation. The 1,891ms p99 cold-start is not a fringe edge case — it occurs after any restart, any scale-to-zero event in serverless mode. Plan pre-warm queries or accept the SLA window.
